# WorkNet MVP PRD（产品需求文档）

## 1. 文档信息
1. 文档版本：v0.3
2. 文档日期：2026-02-26
3. 产品阶段：MVP
4. 适用范围：代码类协作任务（区块链结算）

## 2. 产品背景
在多人协作任务中，平台通常依赖中心化审核来决定“谁贡献了多少”，导致争议高、效率低、可扩展性差。  
本产品通过“贡献声明 + 盲注市场 + 挑战惩罚 + 自动分账”机制，降低搭便车和抄袭，提升协作效率与结果质量。

## 3. 产品目标与非目标

### 3.1 目标（MVP）
1. 支持多人接力提交方案，并形成可追溯贡献图。
2. 通过盲注押注机制输出可用的质量信号。
3. 建立可执行的抄袭挑战与惩罚闭环。
4. 完成奖金与后续收入自动分配的最小闭环。

### 3.2 非目标（MVP 不做）
1. 不在 MVP 阶段推进外部交易所大规模流动性扩张。
2. 不覆盖营销、客服等非技术任务场景。
3. 不做完全去中心化仲裁网络（先采用仲裁委员会）。
4. 不做复杂多路线并行长期运营（每任务仅 1 条胜出路线）。

### 3.3 单币策略（MVP）
1. 发行并使用平台代币 WRK 作为唯一结算、质押与分配资产。
2. 任务预算、保证金、押注与罚没均用 WRK 固定值表达。
3. 参数调整采用固定时间窗（默认每 7 天最多调整一次）。
4. 引入流动性与供给风控，不引入第二记账单位。

## 4. 目标用户与角色
1. Sponsor（发包方）：发布任务、注资、验收并选定胜出路线。
2. Builder（开发者）：提交节点方案，声明原创与引用比例。
3. Backer（押注者）：通过盲注表达对方案质量的判断。
4. Challenger（挑战者）：发起抄袭/未声明引用争议。
5. Arbiter（仲裁者）：审核证据并裁决争议。

## 5. 核心流程（E2E）
1. Sponsor 创建任务并注入 WRK 奖金池（任务预算以 WRK 记账）。
2. Builder 提交方案节点，填写贡献与引用比例（总和=100%）。
3. Backer 在押注窗口进行 commit-reveal 盲注。
4. Sponsor 在候选方案中选定胜出节点。
5. 挑战窗口开启，Challenger 可提交争议并质押保证金。
6. Arbiter 裁决后执行奖池分配、罚没与资格处理。
7. 若任务产出产品后续收入，按固化名单自动分账并注入公共池。

## 6. 信息架构与页面清单（MVP）
1. 页面 A：任务市场（Task List）
2. 页面 B：任务详情与协作图（Task Detail + DAG）
3. 页面 C：提交方案（Submit Node）
4. 页面 D：盲注押注（Commit/Reveal Bet）
5. 页面 E：发包方评审（Sponsor Review）
6. 页面 F：争议与仲裁（Dispute Center）
7. 页面 G：分账与流水（Payout & Revenue）

## 7. 页面级需求

### 7.1 页面 A：任务市场
1. 用户：所有角色
2. 目标：发现并筛选可参与任务
3. 核心信息：
   - 任务标题、预算（WRK）、奖金池（WRK）、状态、截止时间
   - 参与人数、节点数、押注总额
4. 关键操作：进入任务详情、连接钱包
5. 验收标准：
   - 可按状态筛选（进行中/待仲裁/已结算）
   - 点击任务可进入详情页

### 7.2 页面 B：任务详情与协作图
1. 用户：所有角色
2. 目标：理解任务要求与方案演进关系
3. 核心信息：
   - 任务说明、验收标准、奖金规模
   - DAG 节点关系、每节点贡献比例声明
   - 每节点押注热度、提交时间
4. 关键操作：查看节点详情、进入提交/押注/挑战动作
5. 验收标准：
   - 可视化展示父子引用关系
   - 节点详情可看到比例拆分和引用证据哈希

### 7.3 页面 C：提交方案
1. 用户：Builder
2. 目标：提交新节点并完成比例声明
3. 输入项：
   - 节点描述与交付链接
   - 父节点选择（可多选）
   - 原创比例、各父节点引用比例
   - 引用证据（commit hash / 文档 hash）
4. 规则校验：
   - 比例和必须等于 100%
   - 必须支付提交保证金（WRK 固定值）
5. 验收标准：
   - 校验失败不可提交
   - 成功后节点进入 DAG 并可被押注

### 7.4 页面 D：盲注押注
1. 用户：Backer
2. 目标：在不可见环境下注以表达质量判断
3. 阶段：
   - Commit 阶段：提交哈希承诺与金额锁定
   - Reveal 阶段：公开押注目标与随机盐
4. 规则：
   - 开奖前不可见他人押注金额与方向
   - 未 reveal 的押注按规则处理（可配置罚没）
   - 押注最小门槛以 WRK 固定值定义
5. 验收标准：
   - commit 与 reveal 均有链上记录
   - 到期自动关闭并进入下一状态

### 7.5 页面 E：发包方评审
1. 用户：Sponsor
2. 目标：选定胜出节点并触发挑战窗口
3. 核心信息：
   - 候选节点交付质量信息
   - 押注热度参考（揭示后）
   - 风险提示（争议历史、相似度告警）
4. 关键操作：选择胜出节点、确认进入挑战期
5. 验收标准：
   - 单任务仅允许选择 1 个胜出节点
   - 选择后系统进入挑战状态

### 7.6 页面 F：争议与仲裁
1. 用户：Challenger、Arbiter、相关 Builder/Backer
2. 目标：完成抄袭或未声明引用争议的证据提交与裁决
3. 输入项：
   - 挑战类型、证据描述、证据哈希
   - 挑战保证金
4. 裁决结果：
   - 挑战成立：执行违规节点处罚及关联押注罚没
   - 挑战驳回：挑战保证金按规则处理
5. 验收标准：
   - 每个争议有明确状态流转（待审/已裁决/已执行）
   - 裁决后自动触发资金处理

### 7.7 页面 G：分账与流水
1. 用户：所有角色
2. 目标：透明查看奖池分配与后续收入分账
3. 核心信息：
   - 奖池分账明细（Builder、Backer、罚没流向）
   - 后续收入入账记录
   - 建设者分红池 vs 公共资金池比例
4. 验收标准：
   - 每笔分账可追溯到任务与规则
   - 可导出基础流水记录

## 8. 功能需求列表（按优先级）

### 8.1 P0（MVP 必须）
1. 钱包连接与身份识别（地址级）。
2. 任务创建与奖金池注资（WRK）。
3. DAG 节点提交与比例校验（总和=100%）。
4. 提交保证金机制。
5. 盲注押注（commit-reveal）。
6. Sponsor 选定胜出节点。
7. 挑战提交与保证金机制。
8. 仲裁录入与执行结果。
9. 奖池自动分配与链上流水展示。
10. 最低流动性阈值校验（低于阈值限制新任务创建）。

### 8.2 P1（MVP 后增强）
1. 节点风险评分（结合历史挑战数据）。
2. 信誉分系统（地址行为画像）。
3. 任务模板系统（不同默认参数）。
4. 更完善的数据看板与报表导出。

## 9. 业务规则与默认参数（MVP）
1. 结算币种：WRK
2. 节点提交保证金：10 WRK
3. 押注最小金额：20 WRK
4. 挑战保证金：50 WRK
5. 挑战窗口：72 小时
6. 仲裁时限：48 小时
7. 后续收入拆分：建设者 60%，公共池 40%
8. 单节点自占比：不设硬上限（由质押成本、押注市场与争议机制约束）
9. 参数调整周期：默认每 7 天最多调整一次
10. 最低流动性阈值：低于阈值时暂停新任务创建

## 10. 数据与指标（MVP 评估）
1. 任务完成率
2. 平均交付周期
3. 挑战发起率与挑战成功率
4. 押注热度与胜出结果相关性
5. Builder/Backer 次月复参与率
6. 人均争议处理时长

## 11. 风险与合规要求
1. 产品文案避免“保本/收益承诺”等金融承诺表达。
2. 默认术语使用“保证金、分配、协作激励”，避免投资型宣传。
3. AI 检测仅做证据辅助，不作为自动惩罚依据。
4. 所有处罚行为必须经过可审计裁决记录。
5. 代币波动风险需在前台明确披露（预算与结算均为 WRK）。
6. 需预留基础流动性与做市预算，避免奖励结算与质押门槛失真。

## 12. 交付里程碑（产品视角）
1. Sprint 1（第 1-2 周）：任务创建、节点提交、DAG 可视化、基础参数配置
2. Sprint 2（第 3-4 周）：盲注押注闭环、Sponsor 评审与胜出选择
3. Sprint 3（第 5-6 周）：争议与仲裁、奖池分配、流水页面
4. Sprint 4（第 7-8 周）：小范围真实任务灰度、指标回收与参数复盘

## 13. 需后续技术讨论的开放项
1. 仲裁委员会的成员准入与替换机制。
2. 盲注未 reveal 的具体罚没比例。
3. 挑战成功时罚没资金在 Challenger 与公共池间的分配比例。
4. DAG 节点数据上链与链下存储边界。
5. 多链部署顺序与 Gas 成本策略。
6. WRK 的释放节奏与锁仓规则，避免短期抛压破坏博弈激励。
7. 最低流动性阈值的触发条件与恢复条件。
