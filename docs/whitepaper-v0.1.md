# WorkNet 白皮书（MVP v0.1）

## 1. 摘要
WorkNet 是一套基于经济博弈的协作协议。  
它解决的问题是：多人接力开发时，如何在不依赖平台人工审核的前提下，公平分配收益并抑制抄袭、搭便车与低质量参与。

本白皮书定义 MVP 阶段的机制规则，不展开工程实现细节。

## 2. 问题定义
传统协作平台存在四个结构性问题：
1. 贡献不可量化，分账争议高。
2. 方案分支过多，筛选成本高。
3. 抄袭与未声明引用难以惩罚。
4. 任务交付后缺少持续迭代资金闭环。

## 3. 设计目标
1. 让多人接力协作可计算、可追溯、可分账。
2. 让质量筛选由市场承担，而非中心化审核承担。
3. 让反作弊激励可执行，且作弊成本高于收益。
4. 让产品上线后可自我融资、自我迭代。
5. 让占比判定尽可能由去中心化预测形成共识，而非人工主观打分。

## 4. 系统边界（MVP）
1. 单一资产：仅使用 `WRK` 作为结算、质押、罚没与分红资产。
2. 单一场景：仅覆盖代码类协作任务。
3. 单一胜出路线：每个任务仅有 1 条最终胜出节点。
4. 仲裁机制：采用仲裁委员会裁决，链上自动执行结果。

## 5. 角色与对象

### 5.1 角色
1. Sponsor：发布任务、注资奖池、选择最终胜出节点。
2. Builder：提交节点方案并声明分成比例。
3. Backer：对节点押注，提供质量信号并承担错押风险。
4. Challenger：发起抄袭或未声明引用争议。
5. Arbiter：审阅证据并输出裁决。

### 5.2 核心对象
1. Task：任务，包含奖池、时间窗、状态。
2. Node：方案节点，包含交付内容与分成声明。
3. Edge：节点引用关系，表示“当前节点继承父节点价值”。
4. Bet：押注记录，采用 commit-reveal。
5. Dispute：争议记录，包含证据、保证金、裁决结果。
6. Treasury：公共资金池。
7. SharePrediction：Backer 对胜出节点贡献占比的预测向量。

### 5.3 关键术语（权重）
1. 声明权重 `W_i(n*)`：
   - 按 Builder 声明的 `s_self` 与 `s_parent` 递推得到的贡献权重。
   - 它是系统的基础权重，不含占比预测校准。
2. 共识权重 `W_i^*`：
   - Backer 在占比预测窗口提交预测后，经截尾加权均值得到的共识占比。
3. 最终权重 `W_i^final`：
   - 用于最终 Builder 分账的权重。
   - 若占比预测有效，则 `W_i^final = W_i^*`。
   - 若占比预测无效，则 `W_i^final = W_i(n*)`。

## 6. 机制一：协作图与贡献权重计算

### 6.1 节点提交规则
每个新节点必须声明一个分成向量：
1. `s_self`：当前提交者的新增贡献比例。
2. `s_parent_k`：每个父节点的引用比例。
3. 约束：`s_self + Σ s_parent_k = 1`。
4. 不设置单节点自占比硬上限，由质押风险与占比预测共识进行约束。

若约束不成立，节点无效。

### 6.2 贡献权重递推
定义 `W_i(n)` 为地址 `i` 在节点 `n` 的最终贡献权重。  
若 `n` 由地址 `u` 提交，则：

`W_i(n) = s_self(n) * I(i=u) + Σ [ s_parent_k(n) * W_i(parent_k) ]`

其中 `I(i=u)` 为指示函数，成立时为 1，否则为 0。  
该递推使“引用关系”自动转化为“可分账权重”。

### 6.3 直观写法（更易读）
同一个公式可以拆成两条规则：
1. 对当前作者 `u`：
   - `W_u(n) = s_self(n) + Σ [ s_parent_k(n) * W_u(parent_k) ]`
2. 对其他人 `i != u`：
   - `W_i(n) = Σ [ s_parent_k(n) * W_i(parent_k) ]`

含义是：  
当前作者拿到“自己的新增部分”，所有人再按引用比例继承父节点权重。

### 6.4 三节点快速例子
1. 节点 A（Alice）：`s_self=1.0`
2. 节点 B（Bob）：`s_self=0.4, s_A=0.6`
3. 节点 C（Carol）：`s_self=0.5, s_B=0.5`

在 C 上的最终权重：
1. Carol：`W_Carol(C)=0.5`
2. Bob：`W_Bob(C)=0.5*0.4=0.2`
3. Alice：`W_Alice(C)=0.5*0.6=0.3`

因此 C 的价值被分解为：
1. Carol 新增贡献 50%
2. Bob 历史贡献 20%
3. Alice 历史贡献 30%

### 6.5 机制含义
1. Builder 可继承前人成果，不必从零开始。
2. 被后续节点持续引用的早期节点，会持续获得权重累积。
3. 分账不是人工评分，而是以声明递推权重为基线，并由去中心化占比预测做最终校准。

### 6.6 机制一补充：去中心化占比判定（Backer 预测）
在 Sponsor 选出胜出节点 `n*` 后，进入占比预测窗口。

1. 预测对象：
   - 每个有效 Backer `j` 提交一个预测向量 `p_j`，其中 `p_{j,i}` 表示地址 `i` 在 `n*` 的贡献占比。
   - 约束：`p_{j,i} >= 0` 且 `Σ_i p_{j,i} = 1`。
   - 每个地址在同一任务仅允许提交一次预测。
2. 提交方式：
   - 采用 commit-reveal，避免互相抄作业。
   - 未在 reveal 窗口公开的预测视为无效。
3. 共识向量：
   - 设预测权重为 `m_j`（默认取该 Backer 在胜出节点上的有效押注额）。
   - 单地址预测权重上限为 `c`（默认 `c=20%`），超出部分不计入 `m_j`。
   - 对每个地址 `i` 计算截尾加权均值 `TWM_i`（去掉最高/最低各 `q%` 极端预测，默认 `q=10%`）。
   - 归一化得到最终占比向量 `W_i^* = TWM_i / Σ_k TWM_k`。
4. 回退规则：
   - 若有效预测人数小于 `K_min`（默认 5），则回退到声明递推权重 `W_i(n*)`。
5. 作用：
   - 把“占比乱报”问题交给去中心化预测市场处理。
   - 让更接近共识的预测者获得更高收益，形成自校正机制。

## 7. 机制二：盲注市场与质量筛选

### 7.1 盲注流程
1. Commit 阶段：Backer 提交哈希承诺并锁定押注金额。
2. Reveal 阶段：Backer 公布押注目标节点与盐值。
3. 未按时 reveal 的押注按规则罚没。

### 7.2 盐值（salt）说明
1. 盐值是 Backer 自行生成的一段随机字符串，只在 reveal 阶段公开。
2. Commit 时实际承诺的是“押注目标 + 押注金额 + 盐值”的哈希。
3. 盐值的作用是防止他人通过穷举猜测你的押注内容，保证盲注有效。
4. Reveal 时系统用公开内容重新计算哈希，与 commit 记录比对一致才算有效押注。

### 7.3 作用
1. 押注资金形成方案质量信号，降低 Sponsor 筛选成本。
2. 盲注防止大户先手引导，减少跟风行为。
3. 押注者承担风险，迫使其主动做质量审查。

## 8. 机制三：争议、查重与连坐惩罚

### 8.1 争议发起
1. Challenger 支付争议保证金并提交证据。
2. 证据可包含代码对比、提交历史、外部引用证明。
3. AI 检测只作为证据输入，不直接决定处罚。

### 8.2 裁决结果
1. 争议成立：
   - 违规节点失效。
   - 违规 Builder 罚没保证金并失去分配资格。
   - 对违规节点押注的 Backer 按规则罚没（连坐）。
   - Challenger 获得罚没资金中的奖励份额。
2. 争议不成立：
   - Challenger 保证金按规则罚没或部分罚没。

### 8.3 机制含义
1. 处罚 Builder 抑制抄袭。
2. 处罚错误押注者迫使其认真审核，不做盲投。
3. 争议奖励让“抓作弊”成为有经济回报的行为。

## 9. 机制四：结算与持续分红

### 9.1 一次性奖池分配
设任务奖池为 `P`，分配系数为：
1. `α`：Builder 分配池比例。
2. `β`：Backer 分配池比例。
3. `γ`：公共池比例。
4. 约束：`α + β + γ = 1`。

### 9.2 Builder 分配公式
若胜出节点为 `n*`，地址 `i` 的 Builder 收益：

`Reward_builder(i) = α * P * W_i^final`

其中：
1. 若占比预测有效，则 `W_i^final = W_i^*`（来自 6.6 共识向量）。
2. 若占比预测无效，则 `W_i^final = W_i(n*)`（声明递推权重）。

### 9.3 Backer 分配公式
设有效预测 Backer 集合为 `J`（要求已在胜出节点押注且完成有效预测）。对每个 `j in J`，定义误差：

`err_j = ||p_j - W^*||_1 / 2`

其中 `err_j` 取值区间为 `[0,1]`。定义评分：

`score_j = m_j * (1 - err_j)`

则 Backer 收益为：

`Reward_backer(j) = β * P * score_j / Σ_{h in J} score_h`

若 `J` 为空或 `Σ score_h = 0`，则 `β*P` 全部进入公共池。

### 9.4 后续收入分配
任务产出若产生持续收入 `R_t`，按固定比例拆分：
1. `λ` 进入建设者分红池（按固化权重分配）。
2. `1-λ` 进入公共池，用于下一轮悬赏与维护。

## 10. 机制五：参数与治理（MVP）

### 10.1 初始参数
1. 节点提交保证金：`10 WRK`
2. 押注最小金额：`20 WRK`
3. 挑战保证金：`50 WRK`
4. 挑战窗口：`72 小时`
5. 仲裁时限：`48 小时`
6. 奖池分配：`α=0.70, β=0.20, γ=0.10`
7. 后续收入分配：`λ=0.60`
8. 占比预测最小有效人数：`K_min=5`
9. 占比预测截尾比例：`q=10%`
10. 占比预测窗口：`24 小时`
11. 单地址预测权重上限：`c=20%`

### 10.2 参数调整原则
1. MVP 观察期内参数默认冻结。
2. 仅在安全事件或机制失效时触发紧急调整。
3. 每次调整必须公开调整原因、影响范围与生效时间。

## 11. 完整算例（单任务）
设任务奖池 `P=10,000 WRK`。

### 11.1 协作路径
1. 节点 A（Alice）：`s_self=1.0`
2. 节点 B（Bob）：`s_self=0.4, s_A=0.6`
3. 节点 C（Carol）：`s_self=0.5, s_B=0.5`

由递推可得胜出节点 C 的权重：
1. `W_Carol(C)=0.5`
2. `W_Bob(C)=0.5*0.4=0.2`
3. `W_Alice(C)=0.5*0.6=0.3`

### 11.2 占比预测共识
假设有 5 个有效 Backer 完成预测，截尾后得到共识向量：
1. Carol：`W_Carol^*=0.45`
2. Bob：`W_Bob^*=0.25`
3. Alice：`W_Alice^*=0.30`

### 11.3 Builder 奖池分配
1. Builder 池：`αP = 7,000 WRK`
2. Carol：`7,000*0.45 = 3,150 WRK`
3. Bob：`7,000*0.25 = 1,750 WRK`
4. Alice：`7,000*0.30 = 2,100 WRK`

### 11.4 Backer 奖池分配
1. Backer 池：`βP = 2,000 WRK`
2. 假设有效 Backer 评分为：Dave `300`，Eve `500`，Frank `700`
3. 总评分：`1,500`
4. Dave：`2,000*(300/1,500)=400 WRK`
5. Eve：`2,000*(500/1,500)=666.67 WRK`
6. Frank：`2,000*(700/1,500)=933.33 WRK`

### 11.5 公共池
1. 公共池：`γP = 1,000 WRK`
2. 用于下一轮 Bug bounty 与版本迭代。

## 12. 攻击面与约束
1. 垃圾提交攻击：通过提交保证金和最小质量门槛抑制。
2. 盲投攻击：通过押错罚没与 reveal 约束抑制。
3. 抄袭洗稿：通过挑战奖励与连坐惩罚抑制。
4. 串谋裁决：通过裁决记录公开、可审计与委员轮换抑制。
5. 参数滥改：通过观察期冻结与公开变更流程抑制。
6. 占比预测操纵：通过截尾均值、最小预测人数与 commit-reveal 抑制。

## 13. 成功判据（MVP）
1. 任务完成率高于对照组协作模式。
2. 争议成功率能够有效识别违规节点。
3. 押注热度与最终胜出质量呈正相关。
4. Builder 与 Backer 次月复参与率稳定提升。
5. 公共池可支持至少一轮后续迭代任务。

## 14. 路线图
1. 阶段 1：单任务闭环验证（提交-押注-裁决-分配）。
2. 阶段 2：多任务并行与公共池再悬赏。
3. 阶段 3：扩展到营销分佣与客服知识库场景。

## 15. 结语
WorkNet 的核心不是“把协作搬到链上”，而是把协作中的关键矛盾转化为可执行的经济约束。  
在该框架下，贡献、筛选、惩罚、分红形成统一闭环，系统可在最小信任假设下持续运行。
