# 基于经济博弈的去中心化协作与利润分配系统（MVP 文档）

## 1. 文档目标
将产品想法收敛为可落地的最小可行版本（MVP），优先验证机制有效性，而非一次性实现完全去中心化与全领域覆盖。

## 2. MVP 核心验证问题
1. 贡献比例声明机制是否能支持多人接力协作，并减少协作中断。
2. 盲注预测市场是否能有效筛选高质量方案分支。
3. 抄袭挑战与惩罚机制是否能降低未声明引用和白嫖行为。

## 3. MVP 范围边界
1. 场景边界：仅覆盖技术任务（代码类 bounty）。
2. 资产边界：仅使用自发代币 WRK 进行结算与质押。
3. 路线边界：单任务仅允许一条最终胜出路线。
4. 裁决边界：采用“链上执行 + 半中心化仲裁委员会”，AI 仅作为证据辅助。

## 4. 角色定义
1. Sponsor（发包方）：创建任务、注资奖金池、选择最终路线。
2. Builder（开发者）：提交方案节点、声明原创与引用比例。
3. Backer（押注者）：盲注押注潜在胜出节点，提供质量信号。
4. Challenger（挑战者）：对疑似抄袭或未声明引用发起争议。
5. Arbiter（仲裁者）：审查证据并做出裁决（MVP 可白名单）。

## 5. MVP 流程（状态机视角）
1. 任务创建：Sponsor 创建任务并注入奖金池。
2. 节点提交：Builder 提交节点，必须声明父节点引用与比例（总和=100%）。
3. 盲注阶段：Backer 进入 commit-reveal 押注阶段。
4. 路线选择：Sponsor 结合交付质量与市场信号选定胜出节点。
5. 挑战窗口：开放抄袭/未声明引用挑战。
6. 仲裁裁决：依据证据裁定，执行奖池分配与罚没。
7. 后续分红：若产品产生收入，按固化名单自动分账并注入公共池。

## 6. 合约最小集（MVP）
1. `TaskPool`：任务、奖金池、里程碑托管。
2. `ContributionDAG`：方案节点、引用关系、比例校验。
3. `BlindBet`：commit-reveal 盲注机制。
4. `DisputeCourt`：争议流程、质押与罚没执行。
5. `RevenueVault`：后续收入自动分账与公共资金池。

## 7. 关键机制设计

### 7.1 网状协作与比例声明
1. Builder 每次提交新节点必须声明：
   - 自身贡献比例。
   - 各父节点引用比例。
2. 比例约束：所有比例总和必须等于 100%。
3. 引用证据：要求提交 commit hash / 文档 hash 等可追溯信息。

### 7.2 盲注预测市场
1. 采用 commit-reveal 机制，开奖前押注金额与方向不可见。
2. 押注作为质量信号，不直接替代 Sponsor 的最终验收权。

### 7.3 抄袭挑战与连坐惩罚
1. 任意参与者可提交挑战，需支付挑战保证金。
2. AI 相似度分析只提供证据，不自动定罪。
3. 若裁决确认抄袭：
   - 违规 Builder 罚没并失去分配资格。
   - 对违规分支押注的 Backer 按规则罚没（连坐）。

### 7.4 代码即股权与持续运营
1. 胜出路线参与者及有效押注者获得固化分红权。
2. 项目后续收入自动拆分：
   - 建设者分红池。
   - 系统公共资金池（用于后续悬赏和迭代）。

## 8. 反作弊与博弈约束
1. 防贪婪声明：不设单节点自占比硬上限，通过质押成本、押注市场与争议机制约束。
2. 防垃圾提交：节点提交需小额保证金。
3. 防跟风押注：押注必须盲注（commit-reveal）。
4. 防串谋：挑战期 + 挑战成功奖励。
5. 防女巫攻击：地址信誉与质押门槛结合（MVP 先用质押门槛）。

## 9. MVP 默认参数（初始建议）
1. 结算币种：WRK
2. 节点提交保证金：10 WRK
3. 押注最小额度：20 WRK
4. 挑战保证金：50 WRK
5. 挑战窗口：72 小时
6. 仲裁时限：48 小时
7. 收入分配比例：建设者 60%，公共池 40%
8. 参数调整周期：默认每 7 天最多调整一次

## 10. 数据指标与验收标准（4-8 周）
1. 任务完成率（是否高于传统 bounty 模式）。
2. 平均交付周期（从任务创建到选出胜出路线）。
3. 争议率与挑战成功率（抄袭检出效率）。
4. 押注一致性（押注热度与最终胜出的相关性）。
5. 复参与率（Builder 与 Backer 的次月留存）。

## 11. 主要风险与应对
1. 合规风险：避免“投资回报”叙事，强调协作质押与工作分配属性。
2. 仲裁中心化风险：MVP 可白名单，后续逐步过渡到去中心化仲裁网络。
3. 复杂度风险：前端默认参数模板化，隐藏复杂博弈细节。
4. 误伤风险：AI 仅做证据，不做自动处罚；所有处罚需仲裁确认。
5. 代币波动风险：预算与结算统一 WRK，并通过参数调整周期与流动性阈值控制风险。

## 12. 里程碑建议
1. M1（2 周）：完成合约接口草案 + 流程原型（无前端精装）。
2. M2（4 周）：上线测试网闭环（创建任务-提交-押注-裁决-分配）。
3. M3（6-8 周）：引入真实小额任务，跑第一轮真实经济数据。

## 13. 后续扩展（非 MVP）
1. 扩展到营销分佣网络（素材作者 + 渠道方自动分账）。
2. 扩展到客服质押问答与知识库版税复利。
3. 在有稳定现金流后优化 WRK 的治理层设计与长期激励模型。
